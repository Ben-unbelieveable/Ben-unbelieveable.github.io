分布式微调：DDP、DeepSpeed 与 FSDP

发表于 2026-06-30 | 分类于 LLM ， 开发
多卡 LLM 微调的策略选型：DDP 数据并行、DeepSpeed ZeRO、FSDP 全分片及与 LoRA SFT 的配置要点。

单卡 07-01 显存优化 仍不够时，需要分布式微调。本篇从工程角度对比 DDP、DeepSpeed ZeRO、FSDP，说明与 SFTTrainer / LoRA 的组合方式；理论背景见 Math-04/20 §3。

段末注释：DDP（Distributed Data Parallel，分布式数据并行）每卡持有一份完整模型，梯度 AllReduce 同步；FSDP（Fully Sharded Data Parallel，全分片数据并行）将参数/梯度/优化器状态分片到多卡。

系列索引：微调技术路线导读

一、何时需要多卡

场景 建议
7B LoRA SFT 多数单卡可训
7B 全参 SFT 多卡 ZeRO-3 / FSDP
13B+ LoRA 长 context 或多卡 DDP 提吞吐
70B QLoRA 多卡 + DeepSpeed

05-04 全参 vs PEFT：优先 PEFT，多卡常为吞吐而非唯一出路。

二、策略对比

策略 每卡模型 优化器/梯度 通信 典型用途
DDP 完整副本 完整 AllReduce 梯度 LoRA 多卡加速
ZeRO-1 完整 分片 中等 全参省 optimizer
ZeRO-2 完整 分片 + 梯度分片 较高 全参进一步省显存
ZeRO-3 分片 分片 高 大模型全参/QLoRA
FSDP 分片 分片 高 PyTorch 原生大模型

LoRA 微调：可训练参数少，DDP 往往足够（每卡完整基座 + 小 adapter）。
全参微调：ZeRO-2/3 或 FSDP。

六、LoRA 分布式注意点

项 说明
仅 adapter 同步 DDP 梯度 AllReduce 作用于可训练参数
ddp_find_unused_parameters LoRA 冻结基座时通常 False
checkpoint 仅 rank0 保存 adapter，避免重复写
学习率 global batch 变大时可 linear scaling lr（谨慎验证）

八、常见踩坑

现象 对策
多卡 loss 不一致 检查 batch 整除、drop_last
仅 rank0 应 log trainer.is_world_process_zero()
DeepSpeed 与 PEFT 冲突 用 TRL 推荐版本组合；ZeRO-3 加载 adapter 有特殊流程
速度无提升 通信瓶颈；减卡数或增大 micro-batch

九、小结

  • LoRA 微调：优先 DDP 提吞吐。
  • 全参 / 超大模型：ZeRO-2/3 或 FSDP。
  • 与单卡共用同一套数据与 08-01 评估 规范。
